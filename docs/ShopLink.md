# 基于 Redis 构建商户详情 Cache\-Aside 缓存，并结合 TTL 超时剔除兜底，保障缓存与数据库的最终一致性。

## 背景阐述

平台中的商户详情属于典型的读多写少数据，查询频率高，但信息更新频率相对较低。

如果每次查询都直接访问 MySQL，会增加数据库压力。因此，我基于 Redis 构建了商户详情 Cache\-Aside 缓存，通过缓存承担大部分读请求。

## 问题剖析

引入 Redis 缓存以后，虽然降低了数据库压力，但同时带来了缓存和数据库的一致性问题。

例如，商户在后台修改了名称、地址或者营业时间，如果只更新 MySQL，没有及时处理 Redis，用户后续查询时仍然可能读取到旧的商户信息。

因此，商户更新时必须同时考虑数据库数据更新和缓存失效。

但 MySQL 和 Redis 是两个独立的存储系统，普通数据库事务无法同时保证它们的操作原子性，所以这里很难做到强一致性，一般采用最终一致性方案。

## 方案构思

对于缓存更新，我主要考虑了三种方案。

第一种是先删除缓存，再更新数据库。

流程是：

```Plain Text
删除 Redis 缓存
  → 更新 MySQL
```

这种方案存在一个问题：删除缓存后、数据库更新完成前，如果有并发查询进入，就会发现缓存未命中，然后读取数据库中的旧数据，并把旧数据重新写回缓存。

```Plain Text
线程 A 删除缓存
  → 线程 B 查询缓存未命中
  → 线程 B 查询到数据库旧数据
  → 线程 A 更新数据库
  → 线程 B 将旧数据写回缓存
```

这样可能导致缓存中的旧数据一直保留到 TTL 过期，因此我没有选择这个方案。

第二种是同时更新数据库和缓存。

流程是：

```Plain Text
更新 MySQL
  → 更新 Redis
```

这个方案看起来比较直接，但数据库和 Redis 无法放在同一个本地事务中。

可能出现数据库更新成功、Redis 更新失败，或者 Redis 更新成功、数据库事务回滚的问题。

另外，如果两个线程并发更新，还可能因为执行顺序不同，导致旧值最后覆盖新值：

```Plain Text
线程 A 更新数据库为 A
线程 B 更新数据库为 B
线程 B 更新缓存为 B
线程 A 最后更新缓存为 A
```

此时数据库中是新值 B，但缓存中却是旧值 A。

第三种是先更新数据库，再删除缓存。

流程是：

```Plain Text
更新 MySQL
  → 删除 Redis 缓存
```

后续查询发现缓存不存在，就会重新读取数据库中的最新数据，并重新构建缓存。

相比直接更新缓存，删除缓存的操作更加简单，也避免了并发更新时旧值覆盖新值的问题。

## 方案选择

最终我选择了 Cache\-Aside 模式下的“先更新数据库，事务提交后删除缓存”方案。

这里不能只简单地在事务方法中执行：

```Plain Text
更新数据库
  → 删除缓存
```

因为方法执行到删除缓存时，数据库事务可能还没有真正提交。

这时如果有并发查询进入，可能出现：

```Plain Text
事务内修改数据库，但还没有提交
  → 删除 Redis 缓存
  → 并发请求发现缓存未命中
  → 查询到数据库旧数据
  → 将旧数据重新写入缓存
```

另外，如果数据库事务最后回滚，缓存也已经被提前删除，产生了不必要的缓存失效。

因此，我把缓存删除操作放在数据库事务成功提交之后执行。

## 具体实现

在查询商户详情时，采用标准 Cache\-Aside 流程：

```Plain Text
查询 Redis
  → 缓存命中：直接返回
  → 缓存未命中：查询 MySQL
  → 将数据库结果回填 Redis
```

在更新商户信息时：

```Plain Text
开启数据库事务
  → 更新 MySQL 商户数据
  → 发布缓存失效事件
  → 数据库事务成功提交
  → AFTER_COMMIT 监听器执行
  → 删除 Redis 商户缓存
```

核心思想是：缓存失效必须建立在数据库事务已经成功提交的基础上。

如果数据库更新失败或者事务回滚，就不会执行缓存删除，Redis 中原来的数据仍然与数据库保持一致。

如果数据库更新成功，事务提交后删除缓存，下一次查询就会从 MySQL 中读取最新数据并重新写入 Redis。

## 为什么使用事务提交后删除

事务提交后删除缓存主要解决两个问题。

第一，避免事务回滚导致缓存被无效删除。

```Plain Text
数据库事务回滚
  → 不删除缓存
```

第二，避免数据库还没有提交时，缓存已经被删除，导致并发请求读取旧数据并回填缓存。

```Plain Text
数据库成功提交
  → 再删除缓存
  → 后续请求读取数据库新数据
```

因此，相比直接在事务方法中删除缓存，事务提交后删除可以缩小缓存和数据库不一致的时间窗口。

## TTL 兜底机制

即使采用事务提交后删除缓存，仍然要考虑 Redis 删除失败的情况。

例如：

```Plain Text
数据库更新成功
  → Redis 网络异常
  → 缓存删除失败
  → Redis 中暂时保留旧数据
```

所以我还为商户缓存设置了 TTL。

TTL 的作用不是主要的一致性方案，而是异常情况下的兜底机制。如果缓存删除失败，旧缓存也会在 TTL 到期后自动失效，之后请求会重新查询数据库并构建缓存。

因此，两种机制的职责分别是：

- 事务提交后删除缓存：主动、及时地使旧缓存失效。

- TTL 超时剔除：处理删除失败等异常情况，保证数据最终自愈。

## 并发边界

这套方案保证的是最终一致性，而不是强一致性。

在极端并发情况下，仍然可能出现：

```Plain Text
线程 A 缓存未命中
  → 线程 A 查询到数据库旧数据
  → 线程 B 更新数据库并提交
  → 线程 B 删除缓存
  → 线程 A 将旧数据写回缓存
```

此时旧数据可能暂时留在 Redis 中。

不过这种并发窗口比较小，并且缓存设置了 TTL，旧数据最终会自动过期，所以对于商户详情这种读多写少、允许短时间数据延迟的业务场景是可以接受的。

如果业务要求更高，还可以进一步引入延迟双删、缓存版本号、订阅数据库 Binlog 或缓存失效消息补偿机制。

## 复盘总结

总结来说，为了降低商户详情查询对数据库的压力，我基于 Redis 实现了 Cache\-Aside 缓存。

查询时优先读取 Redis，缓存未命中再查询 MySQL并回填缓存；更新时不直接修改缓存，而是先更新数据库，并在数据库事务成功提交后删除缓存。

同时为缓存设置 TTL，作为 Redis 删除失败和极端并发情况下的兜底机制。

最终形成：

```Plain Text
Cache-Aside 负责缓存读取和回填
事务提交后删除负责主动失效
TTL 超时剔除负责异常兜底
```

这套方案不追求缓存与数据库的强一致性，而是在实现复杂度和业务需求之间做出平衡，保障两者的最终一致性。



# **缓存优化**：使用**逻辑过期**方案防止 Redis 热点 Key 的缓存击穿问题；使用**布隆过滤器\+Redis缓存空值**方案协同解决 Redis Key 的缓存穿透问题； 

## 1\.针对缓存击穿

#### a\. 背景阐述

**对于 \(平台的准点开放、限时的一些活动信息\) 或者 \(热搜榜单\)，它们是一个热点数据**，在活动上线之前，我们需要提前预热：将其存入 Redis 中进行缓存，目的是  提高系统的响应速度，降低数据库的访问压力。



#### b\. 问题剖析

针对**高并发场景的热点数据存入 Redis 缓存**，那么我们需要考虑的很重要的一个问题是：缓存击穿的问题

如果缓存的热点数据的TTL失效过期的瞬间，有大量的请求访问，

那么大量请求到达数据库，**导致数据库瞬时压力过大甚至可能崩溃**。



#### c\. 方案构思

因此需要解决缓存击穿的问题，我了解到两种解决方案：一是使用互斥锁。二是使用逻辑过期。

**我先说一下互斥锁的方案是这样的：**

当缓存失效时，有大量请求都会去访问数据库，希望获取数据且重新建立缓存再返回给用户数据。但其实只需要一个请求对应的线程去访问数据库，重新建立起缓存即可，大量的请求去访问数据库重建缓存给数据库带来瞬时压力过大是不必要的。

所以在请求进入后，若缓存不存在，那么需要尝试获取互斥锁，大量的请求中，只有一个线程获取锁成功，由这个线程去查询数据库，重新建立缓存，而其他的请求线程获取锁失败，都休眠一小段时间后不断重试获取缓存数据返回。

**这个互斥锁的方案，最终只有一个线程去访问数据库，降低了数据库的访问压力，解决了缓存击穿的问题，但是其他的线程需要等待，性能受到影响，对于外界可能表现出一些不可用或者延迟的现象，但是保证了强一致性**



**而逻辑过期的方案是这样的：**

针对热点数据我们不设置TTL，那么Redis Key就不会失效，而是在Redis的Value数据中新增一个属性存储过期时间字段。

那么在获取key时需要检查过期时间字段，判断是否过期，如果发现过期，则尝试获取互斥锁，大量的请求中，只有一个线程获取锁成功，由这个线程去开启一个独立线程去查询数据库重建缓存，而本线程就直接返回Redis中的旧数据。而其他的请求线程获取锁失败，都返回Redis中的旧数据。

**这个逻辑过期的方案，最终也是只有一个线程去访问数据库，降低了数据库的访问压力，解决了缓存击穿的问题，并且线程无需等待，性能比较好，保证了服务的可用性，但是部分请求返回了Redis的旧数据，没有保证一致性**



所以这两个方案的对比，其实就是  **可用性和一致性的抉择 \(CAP定理\) **：

- 互斥锁保证了一致性，牺牲了可用性；

- 逻辑过期保证了可用性，而牺牲了一致性

**针对这个\(活动信息\)或\(热搜榜单\)，它其实是一种社交媒体类型的数据，对于一致性的要求并不是强一致性**

**最终我为了保证服务的可用性，选择了使用逻辑过期的方案来解决缓存击穿问题**



在测试阶段，\(手动修改数据库中的数据，使得Redis和MySQL数据不一致\)，然后等到Redis中的数据逻辑上过期后，使用 Jmeter 进行压力测试，在5s内5000个线程并发请求，QPS是1000，查看日志可以发现只有一次查询数据库重建缓存的过程，**证明在高并发的场景下，没有让所有的请求打到数据库，成功使用逻辑过期方案解决缓存击穿问题**。缺点就是在缓存尚未重建完毕时，前面一小部分的请求\(大概200ms是缓存重建的耗时\)获取到的是旧的Redis数据，后面所有的请求获取到的是正确的数据。



#### d\. 复盘总结

所以总结一下，这是一个针对于热点数据、存在高并发请求的场景下的 存入Redis缓存，并且解决缓存击穿问题的一个工作内容。我考虑了常见的两种解决方案，最终**根据应用场景**选择了逻辑过期的方案解决。



#### 细节解释

1. 缓存击穿一定要是针对热点数据，热点数据的特征必须是 高频访问的（如 QPS \> 1000），

    1. 例如：电商的秒杀商品详情、促销活动信息；社交媒体的热门帖子、热搜榜单；实时排行榜（如游戏积分、直播打赏榜单）。

    2. 缓存过期后，大量请求同时涌入数据库。

    3. 这样的数据才值得去解决，所以场景一定要合理。

#### 常见面试题

1. ***问题：你说根据应用场景选择逻辑过期的方案解决缓存击穿，那你觉得什么场景才用互斥锁解决呢？***

**答：**我认为是分业务场景的，例如金融类的业务场景，是要求强一致性的，那么才会使用互斥锁方案，而本项目中其实是一个社交媒体类型的业务场景，所以并不要求强一致性，而选择了服务可用性，采用逻辑过期方案。



2. ***问题：什么是CAP定理***

**CAP 定理**：一个分布式系统最多只能同时满足**一致性\(Consistency\)**、**可用性\(Availability\)**和**分区容错性\(Partition tolerance\) **这三项中的两项。

- **一致性\(C\)**：每次读取都会收到最新的写入数据或错误信息。\(强一致性\)

- **可用性\(A\)**：每个请求都会收到（非错误的）响应，但不能保证响应包含最新的写入数据。

- **分区容错性\(P\)**：发生网络分区或节点故障时，系统仍然能够继续运行。



进一步理解，根据 CAP 定理，分布式系统在网络分区发生时，必须在一致性（C）和可用性（A）之间二选一

- **CP系统**（如ZooKeeper、HBase分布式存储系统）：

    - 分区发生时，优先保证数据一致性。

    - 例如，如果主节点发现自己无法与多数节点通信，会停止写入服务（牺牲可用性A），避免数据不一致。

- **AP系统**（如Redis）：

    - 分区发生时，优先保证可用性。

    - 例如，允许每个分区继续接受读写请求（牺牲一致性C），后续再恢复数据一致。

无论选择CP还是AP，系统都实现了**分区容错性（P）**，即网络分区发生时系统仍能运行，只是策略不同。

**应用场景**：金融、支付分布式系统选择满足CP，社交内容分布式系统选择满足AP。

### **分区容错性\(P）**

分区容错性指当网络发生分区（节点间通信中断）时或者部分节点故障时，系统仍能继续运行，不因部分节点失联而整体崩溃。它是分布式系统的“生存底线”。

### 可用性\(A）

可用性即服务一直可用，且响应时间正常。

一般在衡量一个系统的可用性时，是通过停机时间来计算的

可用水平\(%\)        年可容忍停机时间

99\.9999        \< 1 min

99\.999        \< 5 min

99\.99        \< 53 min

99\.9        \< 8\.8 h

99        \< 3\.65 d

为了实现可用性：

● 冗余设计：多节点部署，避免单点故障。

● 负载均衡：将请求分发到健康节点。

● 熔断机制：故障时快速拒绝请求，防止雪崩。



### 一致性\(C）

在分布式系统中，一致性指所有节点在同一时刻看到的数据是相同的（强一致性）

## 2\.针对缓存穿透

## 背景阐述

平台中的商铺信息变化频率比较低，但是查询频率比较高，属于典型的读多写少数据。

因此，我将商铺详情写入 Redis 缓存。用户根据商铺 ID 查询商铺时，优先读取缓存，缓存未命中再查询 MySQL，并将数据库查询结果回填到 Redis，从而降低数据库压力。

## 问题剖析

但是商铺详情是根据商铺 ID 查询的，如果有人不断使用随机生成、不存在的商铺 ID 发起请求，那么这些数据在 Redis 和 MySQL 中都不存在。

普通 Cache\-Aside 流程下，每个随机 ID 都会经历：

如果攻击者每次使用不同的无效 ID，那么这些请求就会绕过 Redis，不断访问数据库，最终可能导致数据库压力过大。这就是缓存穿透问题。

## 方案构思

针对缓存穿透，我主要了解了两种解决方案：缓存空值和布隆过滤器。

**第一种是缓存空值。**

当 Redis 没有对应的商铺缓存，并且 MySQL 中也不存在该商铺时，就向 Redis 写入一个空字符串，并设置较短的 TTL。



例如：

第一次查询不存在的 ID

→ Redis 未命中

→ MySQL 不存在

→ Redis 缓存空字符串，TTL 2 分钟



再次查询相同 ID

→ 命中 Redis 空值

→ 直接返回店铺不存在



这个方案实现比较简单，可以很好地拦截对同一个无效 ID 的重复请求。虽然会占用额外的 Redis 内存，但是可以通过设置较短的 TTL，让空值自动过期。

不过，如果攻击者每次都使用不同的随机 ID，那么每个 ID 第一次查询时仍然会访问数据库，并且 Redis 中可能产生大量不同的空值键。

**第二种是布隆过滤器。**

应用启动时，我会把数据库中所有真实的商铺 ID 加入布隆过滤器。布隆过滤器会通过多个哈希函数，将商铺 ID 映射到位数组中的多个位置。

收到查询请求后，先判断商铺 ID 是否可能存在：

布隆过滤器的特点是：

- 判断不存在时，数据一定不存在。

- 判断存在时，数据只是可能存在。

- 由于哈希碰撞，存在一定误判率。

- 由于多个数据可能共用相同的 bit 位，数据加入后通常不能直接删除。

因此，布隆过滤器适合快速拦截大量不同的非法 ID，但不能作为最终的数据真实性判断。

## 方案选择

在当前项目中，我没有只选择其中一种方案，而是使用了“布隆过滤器 \+ Redis 缓存空值”的组合方案。

因为这两种方案解决的问题有所不同：

- 布隆过滤器主要拦截大量随机、不同的无效 ID，避免它们访问 Redis 业务缓存和 MySQL。

- 缓存空值主要处理布隆过滤器误判、商铺删除以及同一个无效 ID 被重复查询的情况。

两种方案组合后，可以互相补充。

## 具体实现

项目使用已有的 Redisson 组件，通过 `RBloomFilter<Long>` 保存有效的商铺 ID。

应用启动时，查询数据库中的全部商铺 ID，并写入布隆过滤器。布隆过滤器预计容量设置为 10 万，误判率设置为 1%。



查询链路如下：

收到商铺 ID 查询请求

→ 布隆过滤器判断

→ 一定不存在：直接返回



→ 可能存在：查询 Caffeine 本地缓存

→ Caffeine 未命中：查询 Redis

→ Redis 未命中：查询 MySQL



→ MySQL 存在：回填 Redis 和 Caffeine

→ MySQL 不存在：Redis 缓存空字符串，TTL 2 分钟



布隆过滤器判断“可能存在”，但是数据库中实际不存在时，就是一次误判。这时请求仍然会访问数据库，不过数据库查询为空后，会在 Redis 中缓存空值。

这样，同一个误判 ID 后续再次查询时，就会命中 Redis 空值，不会重复访问数据库。

对于数据维护，我做了两部分处理：

- 应用启动时，从数据库加载全部商铺 ID，对布隆过滤器进行初始化和校准。

- 新增商铺成功后，将新生成的商铺 ID 加入布隆过滤器。

对于删除的商铺，不直接从布隆过滤器删除。因为布隆过滤器的多个数据可能共用同一个 bit 位，随意清除可能导致其他真实数据出现假阴性。删除后的旧 ID 最多产生一次误判，之后会通过 Redis 空值缓存兜底。

另外，如果布隆过滤器初始化失败或者 Redis 出现异常，我采用降级放行策略，继续走原来的缓存和数据库查询链路，避免因为布隆过滤器故障，把真实存在的商铺错误判断为不存在。

## 方案对比

组合方案相比单独缓存空值，主要优势是：

- 能够拦截大量随机、不同的无效 ID。

- 减少无效请求对数据库的访问。

- 减少 Redis 中产生大量空值键的可能。

- 空值缓存可以继续处理布隆过滤器误判。

- 布隆过滤器异常时能够降级，不影响核心查询功能。

它的代价是：

- 增加了布隆过滤器初始化和维护逻辑。

- 存在一定误判率。

- 新增商铺时需要同步更新布隆过滤器。

- 删除数据时不能简单地从布隆过滤器移除。

- 相比单独缓存空值，实现复杂度更高。

## 复盘总结

总结来说，这次优化主要是为了解决商铺详情缓存中的缓存穿透问题。

单独缓存空值只能很好地解决同一个无效 ID 被重复查询的问题，但是面对大量随机、不同的无效 ID时，每个 ID 的第一次请求仍然可能访问数据库。

因此，我在原有 Redis 空值缓存的基础上增加了布隆过滤器。布隆过滤器负责在查询链路前面快速拦截明显不存在的商铺 ID，Redis 空值缓存负责处理布隆过滤器误判和已删除商铺。

最终形成：

这套方案在增加少量实现复杂度的情况下，提高了系统面对随机恶意 ID 请求时的防护能力，并进一步降低了 MySQL 的查询压力。

# **多级缓存**：基于 Caffeine 与 Redis 构建二级缓存，通过 RocketMQ 广播实现集群节点缓存失效，并结合短 TTL 兜底，在保证最终一致性的同时降低 Redis 热 Key 压力。

## 面试话术

### 背景阐述

平台中的优惠券可以分为普通优惠券和秒杀优惠券。当用户进入商户详情页时，需要查询该商户的优惠券列表，其中包含优惠券名称、使用规则、秒杀开始时间和结束时间等信息。

这些静态信息更新频率很低，但在秒杀活动期间访问量很高，属于典型的读多写少热点数据，因此适合使用多级缓存进行优化。

需要特别说明的是，秒杀库存变化频繁，不能作为静态数据缓存在本地。本项目只缓存优惠券静态信息，每次返回结果时再从 Redis 批量读取实时库存并合并。

### 问题剖析

最开始优惠券信息只缓存在 Redis 中。虽然 Redis 的性能较高，但在高并发秒杀场景下，大量用户可能同时访问同一商户的优惠券列表，形成 Redis 热 Key。

如果所有请求都访问 Redis，会增加 Redis 的网络 IO 和序列化开销。当访问量继续增长时，Redis 可能成为系统的性能瓶颈。

因此，我希望把热点数据的访问压力进一步分散到各个应用实例中。

### 方案构思

我的方案是使用 Caffeine 本地缓存和 Redis 缓存搭建二级缓存，查询链路为：

```Plain Text
Caffeine 本地缓存
  -> Redis 缓存
    -> MySQL 数据库
```

具体查询流程如下：

- 首先查询当前服务实例的 Caffeine 本地缓存，命中后直接返回，避免访问 Redis。

- 如果 Caffeine 未命中，则查询 Redis。

- 如果 Redis 命中，则将数据回填到 Caffeine，然后返回。

- 如果 Redis 也未命中，则查询 MySQL，并依次回填 Redis 和 Caffeine。

- 返回数据前，从 Redis 批量查询秒杀优惠券的实时库存，并与缓存中的静态信息合并。

这样，大部分热点查询都可以直接在应用进程内完成，将一个 Redis 热 Key 的访问压力分散到多个应用实例的内存中。

### 数据一致性处理

引入本地缓存以后，集群环境下每个服务实例都会维护自己的 Caffeine 缓存。如果优惠券信息发生更新，只删除 Redis 缓存，其他实例的本地缓存仍然可能保存旧数据。

因此，我采用了“事务提交后删除缓存 \+ RocketMQ 广播失效”的方案：

```Plain Text
更新数据库
  -> 数据库事务提交成功
    -> 删除当前实例的 Caffeine
    -> 删除 Redis 缓存
    -> 发送 RocketMQ 广播消息
      -> 所有服务实例删除各自的 Caffeine
```

RocketMQ 消费者采用广播消费模式，因此同一个消费者组中的每个应用实例都能收到缓存失效消息。

这里发送的是“缓存失效消息”，而不是直接把新数据广播给所有节点。这样可以避免消息乱序导致旧数据覆盖新数据，后续请求会按照 Cache\-Aside 流程重新加载最新数据。

同时，我给 Caffeine 设置了较短的 TTL，给 Redis 设置了相对较长的 TTL。即使 RocketMQ 消息发送失败或某个实例暂时没有收到消息，旧数据也会在 TTL 到期后自动失效，作为最终一致性的兜底。

### 为什么不只使用短 TTL

只使用短 TTL 的实现确实更简单，但是会存在一段固定时间的数据不一致窗口，而且本地缓存会频繁失效并重新访问 Redis。

当前项目已经使用 RocketMQ，因此增加广播缓存失效消息的成本不高。综合考虑后，我选择：

- RocketMQ 广播：主动保证各实例及时失效缓存。

- Caffeine 短 TTL：处理消息丢失或消费失败等异常情况。

- Redis TTL：作为共享缓存的最终过期兜底。

两者结合，比单独依赖短 TTL 的一致性和实时性更好。

### 复盘总结

总结一下，针对秒杀优惠券静态信息访问频率高、更新频率低的特点，我使用 Caffeine 和 Redis 搭建了二级缓存。

Caffeine 将 Redis 热 Key 的访问压力分散到各个应用实例；Redis 作为集群共享的二级缓存；MySQL 作为最终数据源。

在一致性方面，数据库事务提交后删除缓存，并通过 RocketMQ 广播通知所有实例清理本地缓存，同时使用短 TTL 兜底。对于频繁变化的秒杀库存，则不进入本地缓存，而是每次从 Redis 实时读取，避免静态缓存影响秒杀库存的准确性。

# 秒杀防超卖和一人一单：基于 Redis \+ Lua 原子完成库存预扣和用户购买资格校验，结合 RocketMQ 异步下单、Redisson 用户锁及 MySQL 唯一索引兜底，保证高并发下库存不超卖及用户对同一优惠券只能下单一次。

### 背景阐述

平台支持商家发布限量秒杀优惠券，秒杀开始时会有大量用户同时请求。系统需要保证两个核心问题：

第一，库存不能被超卖；第二，同一个用户对同一张优惠券只能购买一次。

### 问题剖析

最直接的方案是每次请求都查询 MySQL，判断用户是否已经购买过优惠券，同时通过分布式锁保证并发安全；扣库存时执行：

```Plain Text
update tb_seckill_voucher
set stock = stock - 1
where voucher_id = ? and stock > 0
```

这种方案虽然能够保证数据正确，但高并发下大量请求会集中访问 MySQL，并且分布式锁和数据库行锁会产生竞争，数据库压力比较大，接口响应时间也会受到影响。

### 方案构思

因此，我将高并发下的下单资格判断放到了 Redis 中。

Redis 中主要保存两类数据：

- `seckill:stock:{voucherId}`：使用 String 保存秒杀库存；

- `seckill:order:{voucherId}`：使用 Set 保存已经抢购过该优惠券的用户 ID。

请求进入后，通过 Lua 脚本原子完成以下操作：

1. 判断 Redis 库存是否大于 0；

2. 判断当前用户是否已经存在于购买用户 Set 中；

3. 校验通过后，将库存减 1，并将用户 ID 加入 Set。

Redis 里的库存扣减和用户记录是在同一个 Lua 脚本中完成的，因此并发请求下不会出现“多个请求同时判断通过”的问题。

### 订单落库

Redis 校验成功后，接口不会同步等待数据库创建订单，而是将订单消息发送到 RocketMQ，由消费者异步处理。

消费者侧又增加了三层保护：

- 使用 Redisson 用户锁，串行处理同一用户的订单消息；

- 创建订单前再次查询 `user_id + voucher_id` 是否已有订单；

- MySQL 的 `tb_voucher_order` 表增加唯一索引：

```Plain Text
unique key uk_user_voucher(user_id, voucher_id)
```

因此，Redis 负责高并发下的快速拦截，MySQL 唯一索引负责最终兜底，即使发生重复消息或并发消费，也不会生成第二笔订单。

### 防超卖保障

Redis Lua 脚本会先预扣库存，防止大量请求直接冲击数据库；订单真正落库时，MySQL 仍然使用：

```Plain Text
stock = stock - 1
where voucher_id = ? and stock > 0
```

作为数据库层的库存保护。

所以整体上是：

```Plain Text
Redis + Lua：快速判断资格并预扣库存
RocketMQ：异步削峰、解耦订单创建
Redisson：串行处理同一用户消息
MySQL 条件扣库存：防止数据库库存变负
MySQL 唯一索引：最终保证一人一单
```

### 复盘总结

这个项目没有单纯依赖数据库查询和分布式锁，而是将高并发下的资格判断前移到 Redis，通过 Lua 保证库存扣减和一人一单判断的原子性，再使用 RocketMQ 异步创建订单，最后通过 MySQL 唯一索引和条件扣库存进行兜底，从而同时保证了秒杀场景下的性能和数据正确性。

# 秒杀流程优化：使用RocketMQ改同步流程为异步流程，实现未支付订单到期自动关单及秒杀链路的削峰解耦。

### 背景阐述

平台上商家可以发布限量秒杀优惠券，秒杀开始时会有大量用户同时发起请求。

在这个场景下，系统不仅需要保证库存不超卖和一人一单，还需要解决订单创建后的支付问题：用户下单成功后，如果在规定时间内没有完成支付，就需要自动关闭订单，并释放 MySQL 和 Redis 中占用的库存。

### 问题剖析

原来的同步流程是：

```Plain Text
用户请求
  -> 校验库存和一人一单
  -> 扣减数据库库存
  -> 创建订单
  -> 等待用户支付
```

如果所有操作都在用户请求线程中同步完成，会导致大量请求同时访问 MySQL，竞争库存行锁和数据库连接，数据库容易成为秒杀链路的瓶颈。

另外，订单创建成功后，还需要持续检测用户是否支付。如果采用不合理的超时检测方式，可能会出现数据库压力过大、订单误关闭或者库存没有正确释放的问题。

### 方案构思

因此，我将秒杀流程拆分成两个异步处理阶段：

第一阶段，通过 Redis \+ Lua 完成高并发下的库存校验、一人一单校验和 Redis 库存预扣。

第二阶段，通过 RocketMQ 异步完成 MySQL 库存扣减和订单创建，避免用户请求同步等待数据库操作。

订单创建成功后，再通过 RocketMQ 延迟消息触发超时检查。如果订单仍然未支付，则自动关闭订单并释放库存。

### 秒杀请求异步化

用户请求进入后，首先执行 Redis Lua 脚本：

```Plain Text
1. 判断 Redis 库存是否大于 0；
2. 判断当前用户是否已经购买过该优惠券；
3. 校验通过后，将 Redis 库存减 1；
4. 将用户 ID 加入已购买用户集合。
```

Redis 校验成功后，系统生成订单 ID，并将订单消息写入本地消息表，再通过 RocketMQ 异步发送订单消息。

接口不再同步等待订单真正落库，而是立即返回：

```Plain Text
订单 ID + 排队中状态
```

前端拿到订单 ID 后，通过订单状态接口轮询后续处理结果。

优化后的秒杀流程是：

```Plain Text
用户请求
  -> Redis + Lua 校验资格并预扣库存
  -> 写入本地消息表
  -> RocketMQ 异步发送订单消息
  -> 接口立即返回排队中
  -> 消费者异步扣减 MySQL 库存并创建订单
  -> 前端轮询订单状态
```

这里需要说明的是，Redis 库存是在请求阶段同步预扣的，MySQL 库存扣减和订单创建是在 RocketMQ 消费者中异步完成的。

### RocketMQ 消费者创建订单

消费者收到秒杀订单消息后，使用 Redisson 用户锁保证同一用户的订单消息串行处理。

之后在事务中完成：

```Plain Text
1. 查询用户是否已经存在该优惠券订单；
2. 使用 stock > 0 条件扣减 MySQL 库存；
3. 创建待支付订单；
4. 发送订单超时延迟消息。
```

如果 RocketMQ 发生重复投递，消费者会通过已有订单查询和 MySQL 的：

```Plain Text
unique(user_id, voucher_id)
```

唯一索引保证同一用户不会创建重复订单。

### 超时未支付订单自动关单

在订单创建成功后，系统会向 RocketMQ 发送一条订单超时延迟消息，当前项目配置为延迟约 5 分钟。

延迟消息到期后，由订单超时消费者执行：

```Plain Text
1. 根据订单 ID 查询订单；
2. 只有订单状态仍然是未支付时，才更新为已取消；
3. 回补 MySQL 库存；
4. 执行 Redis 回滚脚本，恢复 Redis 库存并移除用户购买资格。
```

核心更新条件是：

```Plain Text
update tb_voucher_order
set status = '已取消'
where id = ?
  and status = '未支付'
```

这样可以保证只有未支付订单才能被关闭。

如果用户已经支付成功，延迟消息仍然会正常到达，但是由于订单状态已经不是未支付，更新操作影响行数为 0，消费者直接返回，不会关闭订单，也不会重复回补库存。

这是一种基于订单状态判断的幂等处理方式。

### 超时关单方案选型

针对超时未支付订单，我调研过几种常见方案。

#### 方案一：定时任务扫描 MySQL

每分钟扫描一次：

```Plain Text
select *
from tb_voucher_order
where status = '未支付'
  and create_time < 当前时间 - 5分钟
```

然后关闭超时订单并释放库存。

这种方案实现简单，但订单量较大时，定时任务会周期性扫描数据库，可能增加数据库压力，而且关单时间会受到定时任务执行周期影响。

#### 方案二：RocketMQ 延迟消息

订单创建成功后发送一条延迟消息，5 分钟后由消费者检查订单状态并执行关单。

这种方案比较适合当前项目，因为 RocketMQ 本身支持延迟消息、消费重试和死信队列，订单创建和订单超时处理也可以通过消息进行解耦，不需要定时扫描整个订单表。

#### 方案三：Redis 过期监听

为每个订单设置带 TTL 的 Redis Key，监听 Key 过期事件后执行关单。

这种方案虽然速度较快，但 Redis 过期通知不是严格准时的，受到 Redis 惰性删除和定期删除机制影响，不能保证订单在精确时间被处理，因此当前项目没有采用这种方案。

综合考虑当前项目是单体项目，订单量相对有限，但又已经使用 RocketMQ 处理订单消息，因此最终选择了 RocketMQ 延迟消息方案。

### 消息可靠性处理

秒杀订单消息使用本地消息表记录发送状态、重试次数和下次重试时间。如果订单消息发送失败，系统会通过定时任务进行补偿重试。

超时关单消息由 RocketMQ 消费者处理，消费者异常时会抛出异常，让 RocketMQ 进行重新投递。当前项目配置了最大重试次数，超过重试次数后进入死信队列，并由死信消费者记录异常消息，方便后续排查。

当前版本的死信消息主要用于异常记录，后续还可以继续扩展自动补偿或人工补偿能力。

### 复盘总结

所以，这次秒杀流程优化主要解决了两个问题：

第一，通过 Redis \+ Lua 和 RocketMQ 异步下单，减少用户请求对 MySQL 的同步依赖，利用消息队列削峰填谷，提高秒杀场景下的并发处理能力。

第二，通过 RocketMQ 延迟消息实现订单超时自动关单，消费者只关闭仍然处于未支付状态的订单，并在关闭成功后回补 MySQL 和 Redis 库存，避免订单超时后库存长期被占用。

整个流程可以总结为：

```Plain Text
Redis + Lua：校验资格并预扣库存
RocketMQ：异步创建订单和发送延迟关单消息
消费者事务：扣减 MySQL 库存并创建订单
延迟消费者：检查未支付订单并自动关闭
状态条件更新：保证关单操作幂等
Redis + MySQL 回补：释放超时订单占用的库存
本地消息表和重试：保证订单消息可靠投递
```

# **滑动窗口限流**： 使用 Redis \+ AOP \+ 注解实现限流，支持全局、IP、用户多维度，防止系统过载、刷券、爬虫；

### 背景阐述

我在项目里考虑了几类高风险接口。

第一，秒杀接口在活动开始时会出现短时间的大量请求，如果不做限制，流量会继续进入 Redis、RocketMQ 和后续数据库链路，影响系统可用性。

第二，验证码和登录接口容易被恶意调用，可能造成短信资源浪费或密码撞库风险。

第三，商户搜索、热门笔记等查询接口容易被爬虫高频访问；图片上传、发布笔记这类写接口也需要限制频率，避免资源被滥用。

所以我实现了一个可复用的限流组件，而不是把限流逻辑散落在各个 Controller 中。

### 问题剖析

限流不能只看“接口总请求数”。例如秒杀接口既需要限制整体流量，避免系统过载；也需要按 IP 限制，避免单个 IP 高频刷请求；还需要按登录用户限制，避免同一个用户频繁提交秒杀请求。

因此我希望这个组件具备几个特点：

- 无侵入：业务代码不需要手写 Redis 限流逻辑；

- 可配置：不同接口可以配置不同维度、窗口和阈值；

- 分布式可用：多实例部署时，限流数据不能只保存在单机内存；

- 原子性：多个维度的判断和计数不能出现并发竞争。

### 方案构思

无侵入和易用性方面，我使用 AOP \+ 自定义注解；高性能和分布式共享方面，我使用 Redis；原子性则通过 Lua 脚本保证。

我调研过几种常见算法：

1. 固定窗口实现简单，但存在临界点问题。例如一分钟结束前放行 100 个请求、下一分钟开始又放行 100 个请求，极短时间内可能放过接近两倍流量。

2. 漏桶算法可以平滑流量，但请求只能按固定速率被处理。秒杀这种突发场景下，即使系统有处理能力，也会额外增加排队或丢弃请求。

3. 令牌桶允许一定突发流量，更适合网关带宽控制；但当前项目更关注“任意连续一分钟内”的严格访问上限，例如限制单 IP 爬虫、单用户频繁提交秒杀，因此不希望令牌积攒后被一次性消耗。

最终我选择了滑动窗口算法。它统计的是最近一段连续时间内的真实请求数，可以避免固定窗口的临界突发问题，更适合当前项目的接口防刷和保护场景。

### 注解与 AOP 实现

我定义了 `@RateLimit` 注解，每个接口可以配置多个 `LimitRule`。每个规则包含：

```Plain Text
限流维度：GLOBAL / IP / USER
时间窗口：windowSeconds
最大请求数：limit
资源标识：key
```

AOP 切面会拦截所有带 `@RateLimit` 的接口，读取当前用户和客户端 IP，组装出对应的 Redis Key，再调用 Lua 脚本执行限流判断。

例如当前秒杀接口配置为：

```Plain Text
全局：60 秒最多 300 次
IP：60 秒最多 30 次
用户：60 秒最多 5 次
```

只有全局、IP、用户三个维度都没有超限时，请求才会进入秒杀业务。

验证码接口则只配置全局和 IP 维度，因为用户在请求验证码时还没有登录；商户搜索和热门笔记接口主要按 IP 限制，用于降低爬虫压力。

### Lua 脚本实现

滑动窗口使用 Redis 的 ZSet 实现：

```Plain Text
Key：限流资源 + 限流维度 + 标识，例如：
rate:sliding:{voucher-seckill}:user:1001

Score：请求发生的毫秒时间戳
Member：UUID，保证同一毫秒内多个请求也不会覆盖
```

Lua 脚本的执行过程是：

```Plain Text
1. 删除时间窗口外的旧请求记录；
2. 使用 ZCARD 统计当前窗口内的请求数；
3. 如果任意一个维度已经达到阈值，直接拒绝本次请求；
4. 如果所有维度都未超限，则向每个维度对应的 ZSet 写入本次请求；
5. 为 ZSet 设置过期时间，避免历史限流 Key 长期占用内存。
```

整个过程在一段 Lua 脚本中完成，所以全局、IP、用户多维度的校验和计数是原子的，不会出现多个并发请求同时判断“未超限”后都被放行的问题。

另外，Redis Key 使用了 Hash Tag，例如：

```Plain Text
rate:sliding:{voucher-seckill}:global:all
rate:sliding:{voucher-seckill}:ip:127.0.0.1
rate:sliding:{voucher-seckill}:user:1001
```

这样即使后续使用 Redis Cluster，同一个接口的多维 Key 也会落在同一个槽位，Lua 脚本可以正常一次操作多个 Key。

### 限流后的响应

当任意维度超限时，切面会抛出限流异常，由统一异常处理器返回：

```Plain Text
HTTP 429
Retry-After：建议等待秒数
Result.fail("操作过于频繁，请稍后再试")
```

前端或调用方能够明确知道这是限流，而不是普通业务异常。

### 复盘总结

这个限流模块的核心不是“为了使用 Redis 或 Lua”，而是针对秒杀防流量冲击、验证码与登录防刷、搜索接口防爬、上传和发布接口保护等实际场景做的统一治理。

最终我通过 Redis ZSet 记录请求时间戳，Lua 保证多维滑动窗口判断的原子性，AOP 和注解保证接入无侵入；业务接口只需要添加注解并配置规则，就可以按全局、IP、用户维度获得统一限流能力。
